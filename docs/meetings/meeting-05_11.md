## 논의 내용

### 1. 유스케이스 다이어그램 AI 초안 검토 및 이름 수정

- 분석가가 Claude로 생성한 유스케이스 다이어그램 초안을 공유하였다.
- 팀 전체가 유스케이스 이름과 시스템 경계명을 검토하면서 아래 의견이 나왔다.
  - "신뢰도 보기"는 요구사항 정의서 표현과 달라 나중에 혼동될 수 있으므로 표현을 통일해야 한다
  - "오류 신고하기"는 실제 기능(틀린 분석 결과에 대한 피드백 전송)과 느낌이 다르므로 수정이 필요하다
  - 시스템 경계명을 "TrueFilter 시스템"으로 변경해야 공식 문서 표현에 맞다
  - "API 호출하기"는 구현 관점 표현이므로 사용자·기능 관점 이름으로 바꿔야 한다

- 팀 논의 결과 아래와 같이 이름 수정 방향을 확정하고, 가이드 "동사 + 목적어" 형식 충족 여부를 AI 교차검증으로 확인하였다.

| AI 원본 | 수정 후 | 수정 이유 |
|---------|---------|-----------|
| 오버레이 켜기 | 오버레이 활성화하기 | 기능 의미 명확화 |
| 신뢰도 보기 | 신뢰도 확인하기 | 요구사항 정의서 표현과 통일 |
| 편향도 보기 | 편향도 분석 확인하기 | 기획서 표현과 통일 |
| 화면 설정하기 | 오버레이 설정하기 | 대상 명확화 |
| 오류 신고하기 | 피드백 전송하기 | 실제 기능 의미와 일치 |
| 텍스트 읽기 | 텍스트 인식하기 | 기능 의미 명확화 |
| API 호출하기 | 팩트체크 API 조회하기 | 구현 관점 → 기능 관점으로 개선 |
| 기사 더 보기 | 유사 기사 조회하기 | 요구사항 정의서 표현과 통일 |
| TrueFilter (경계명) | TrueFilter 시스템 | 공식 문서 표현 통일 |

- AI 교차검증 결과: 수정된 이름 전체가 가이드 기준에 적합하다고 확인됨. 구조와 관계는 AI 원본 유지하기로 결정.

### 2. UC-02 유스케이스 설명서 예외 처리 구체화

- 설계자(최라혜)가 Claude로 생성한 UC-02 설명서 초안의 예외 처리 부분을 팀이 검토하였다.
- AI 원본 예외 처리가 결과만 적혀 있고 구체적인 동작이 빠져 있다는 의견이 나왔다.
  - API 오류 시 "오류 메시지 표시"만 적혀 있어 어떤 메시지를 어떻게 표시하는지 알 수 없다
  - 텍스트 인식 실패 기준이 모호하며, 팀이 합의한 30자 이상 조건이 반영되어야 한다
  - API 오류 시 결과를 전혀 표시하지 않는 것이 아니라 부분 계산 후 표시하는 방식으로 명시해야 한다

- 팀 논의로 아래와 같이 예외 처리를 구체화하였다.

| 식별자 | 예외 상황 | 수정 후 처리 |
|--------|-----------|-------------|
| S4-1a | API 응답 2초 초과 | 타임아웃 처리 후 "분석 중" 표시, 결과 도착 시 업데이트 |
| S2-1a | 인식 텍스트 30자 미만 | 오버레이 미표시 |
| S4-2a | API HTTP 5xx 오류 | 출처·인용 점수만으로 부분 계산 후 "팩트체크 미포함" 표시 |

- AI 교차검증 결과: 정상 시나리오와 충돌 없음 확인. AI가 TTL 초과 후 재분석 케이스 누락을 추가로 지적하였으나, 팀 논의 결과 M3 구현 단계에서 다루기로 하고 이번 설명서 범위에서 제외하기로 결정.
