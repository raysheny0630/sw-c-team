## 1. 프로젝트 개요

**프로젝트명:** TrueFilter

### 배경 및 문제 정의

SNS 혹은 인터넷 뉴스 사용 중 알고리즘이 자극적이고 왜곡된 기사만 반복 노출시키는 것이 문제라고 느꼈다.  
페르소나 인터뷰(단국이, 20세)에서 다음 두 가지 답변을 받았다.

- "잘못된 정보를 사실인 줄 알고 얘기했다가 민망했다"
- "따로 검증하는 작업은 번거로우므로 잘 하지 않는다"

### 목적

SNS 피드·뉴스를 보는 도중 앱을 전환하지 않고도 **오버레이 방식**으로 신뢰도·편향도 분석 결과를 즉시 볼 수 있게 하여, 사용자가 스스로 정보를 판단할 수 있도록 돕는다.  
AI 인터뷰에서 *"판단해주는 느낌이면 거부감이 들 것 같다, 근거를 같이 보여주면 신뢰할 것"* 이라는 피드백을 받았으므로, 근거(출처 공신력·유사 기사 수·팩트체크 결과)를 함께 제공한다.

### 예상 사용자 및 플랫폼

| 항목 | 내용 |
|------|------|
| 예상 사용자 | SNS에서 시사 정보를 주로 접하는 사용자 |
| 플랫폼 | Android / iOS |

### 주요 기능 요약

1. SNS 화면 텍스트 실시간 인식
2. 데이터 기반 신뢰도 5단계 척도 계산
3. 유사 기사 리스트 제공 (최대 5건) + 편향도 1문장 요약
4. 오버레이 창 사용자 편의 기능 (위치·투명도 조절)
5. 사용자 피드백 전송 기능

## 3. 요구사항 정의

아래 요구사항은 5주차 페르소나 인터뷰 결과를 정제하여 도출하였다.

### 기능 요구사항 (Functional Requirements)

| ID | 요구사항 내용 | 우선순위 |
|----|--------------|---------|
| FR-01 | 시스템은 SNS 앱 실행 중 화면상에 표시된 텍스트(뉴스 제목·본문 일부)를 실시간으로 인식할 수 있어야 한다. | 상 |
| FR-02 | 시스템은 인식된 텍스트에 대해 출처 공신력·팩트체크 결과·인용 여부를 합산한 데이터 기반 신뢰도 점수를 5단계 척도로 산출할 수 있어야 한다. *(근거: 주관적 판단 배제, AI 환각 리스크 최소화)* | 상 |
| FR-03 | 사용자는 오버레이 창의 화면 위치(드래그)와 투명도(슬라이더)를 직접 조절할 수 있어야 한다. *(근거: 텍스트 영역 침범 방지)* | 중 |
| FR-04 | 시스템은 편향도 분석 결과를 1문장으로 요약하고, 사용자 요청 시 근거가 된 유사 기사 리스트(최대 5건)를 제공할 수 있어야 한다. *(근거: AI 인터뷰 Q5 — 근거를 같이 보여줘야 신뢰한다는 피드백 반영, 본 앱의 핵심 차별점)* | 중 |
| FR-05 | 사용자는 시스템이 오분석 결과를 제공했을 때 수정을 요청하는 피드백을 앱 내에서 전송할 수 있어야 한다. | 하 |

### 비기능 요구사항 (Non-Functional Requirements)

| ID | 요구사항 내용 | 품질 특성 | 우선순위 |
|----|--------------|----------|---------|
| NFR-01 | 시스템은 텍스트 인식 완료 후 오버레이 결과 출력까지 2초를 초과해서는 안 된다. *(기준: Nielsen 1993 응답시간 가이드라인 — 사용자 인지 지연 임계값 2~3초)* | 성능 | 중 |
| NFR-02 | 시스템은 사용자의 SNS 화면 텍스트 인식 데이터를 기기 외부로 전송할 경우 TLS 1.3 이상으로 암호화하여야 하며, 분석 완료 후 원문은 즉시 파기해야 한다. | 보안 | 중 |
| NFR-03 | 사용자는 별도의 가이드북 없이도 처음 앱을 설치한 후 3분 이내에 오버레이 활성화 기능을 스스로 실행할 수 있어야 한다. | 사용성 | 하 |

## 5. 비용 산정 (간이 FP)

요구사항 목록(FR-01~FR-05)을 기반으로 간이 기능 점수(FP)를 산정한다.  
※ NFR(비기능 요구사항)은 FP 산정 대상이 아니다.

| 기능 유형 | 해당 기능 | 개수 | 가중치 | FP |
|----------|----------|:----:|:------:|:---:|
| EI (외부 입력) | FR-01 텍스트 인식, FR-03 오버레이 위치·투명도 설정 입력, FR-05 피드백 전송 | 3 | 3 | 9 |
| EO (외부 출력) | FR-02 신뢰도 5단계 척도 오버레이 출력 | 1 | 4 | 4 |
| EQ (외부 조회) | FR-04 편향도 근거·유사 기사 리스트 조회 | 1 | 3 | 3 |
| ILF (내부 논리 파일) | 사용자 오버레이 설정 정보 (FR-03 지원), 분석 결과 캐시 (FR-01·FR-02 중복 요청 방지) | 2 | 7 | 14 |
| EIF (외부 인터페이스 파일) | 외부 뉴스 신뢰도 DB (FR-02·FR-04 지원) | 1 | 5 | 5 |
| **합계** | | **8** | | **35 FP** |

> **ILF 설명:** ILF는 기능 요구사항을 직접 지원하기 위해 시스템 내부에서 생성·관리하는 데이터 파일이다.  
> - *사용자 오버레이 설정 정보*: FR-03(위치·투명도 조절)의 설정값을 앱 재실행 시에도 유지하기 위해 내부 저장소에 관리한다.  
> - *분석 결과 캐시*: FR-01·FR-02에서 동일 텍스트를 반복 인식할 경우 중복 API 호출을 방지하기 위해 최근 분석 결과를 내부적으로 캐싱한다.

### 공수 산정

| 항목 | 값 |
|------|----|
| 총 기능 점수 | 35 FP |
| 팀 생산성 (학부 프로젝트 기준) | 12 FP/인월 |
| 필요 공수 | 35 ÷ 12 ≈ **2.92인월** |

## 6. AI 활용 계획

### 활용 도구

Claude, ChatGPT, Gemini 등 각 팀원이 사용하는 도구를 활용하며, 복수 도구로 교차 검증한다.

### 단계별 AI 활용 방침

| 단계 | AI 활용 방식 | 검토 기준 |
|------|------------|---------|
| M1 기획 | 페르소나 생성, 요구사항 초안 도출 | 치명적 과실 유형 자가 점검 후 팀 정제 |
| M2 설계 | 유스케이스·클래스 다이어그램 초안 작성 | 설계자의 수정 + 팀 전원 검토 |
| M3 구현 | 시스템 로직·코드 작성 지원 | 개발자가 직접 검증 |
| M3 검토 | 보안 취약점 목록 열거 지원 | QA/보안 담당자가 OWASP 기준으로 검토 |

### AI 활용 3원칙 준수 방침

| 원칙 | 내용 |
|------|------|
| 단순 복사 금지 | AI 생성 결과물을 그대로 제출하지 않는다 |
| 비판적 검증 | AI 답변의 논리적·보안적 결함을 반드시 확인한다 |
| 수정 이력 명시 | 프롬프트 기록과 수정/보완 과정을 개인 AI 로그에 구체적으로 작성한다 |

### 크로스 체크 (Mutual Review) 운영 방식

13주차에 팀 내 역할 교환 방식으로 Cross-check를 수행한다. 팀 간 상호 검토는 진행하지 않는다.

| 검토 방향 | 검토자 | 검토 내용 |
|----------|--------|---------|
| 분석가 산출물 → | 개발자 | 구현 로직이 초기 요구사항과 일치하는지 추적성 검증 |
| 설계자 산출물 → | QA/보안 | 구조적 결함 및 모듈화 수준 검토 |
| 개발자 산출물 → | 분석가 | 요구사항의 기술적 실현 가능성 점검 |
| QA/보안 산출물 → | 설계자 | 보안 취약점 스캔 및 인스펙션 |
