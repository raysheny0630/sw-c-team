**프롬프트 (입력한 내용 요약)**

403 인증 오류가 반복되는 상황에서 오류 로그를 첨부하고 원인 진단과 해결 방법을 단계적으로 요청함. (개발자와 같은 세션에서 공동 진행)

**AI 응답 요약**

환경 변수 삭제 → `%APPDATA%\gcloud` 자격 증명 파일 삭제 → 환경 변수 직접 삭제 순으로 여러 해결책을 제시했으나, 각 방법의 전제 조건이 실제 PC 환경과 달라 반복 실패함. 이후 라이브러리를 우회하는 REST API 직접 호출 방식을 제시함.

**비판적 검토 (PM 관점)**

- 일정 관점에서 무한 시도는 불가능하므로, 원인 계층(라이브러리 내부 인증 우선순위)을 우회하는 구조적 해법(REST 직접 호출)을 채택 기준으로 잡아야 했음.

**팀/개인 수정 내용**

- "라이브러리 의존 제거(REST 직접 호출)" 방향을 채택 기준으로 확정하고, 모델명 404 반복에 대해서는 추측 시도 대신 사용 가능 모델 목록을 먼저 조회하는 절차를 다음 규칙으로 정함 (상세 기술 수정 내역은 개발자 로그 참조).
- 이 경험을 M3 보고서 §12-5 "AI 활용의 한계 사례"로 정리함.

**최종 활용 결과**

`TrueFilter/main.py`의 REST 직접 호출 전환 완료, 프로토타입 12주차 계획 내 완료 (WBS 작업 7·8 정시 완료에 기여).

| 관련 주차 / 활동 | 15주차 — M3 보고서의 SW공학 쟁점 처리 방향 자문 (6/11 회의 안건 2) |
| 사용 도구 | Claude |
| 역할 연관성 | 팀 자체적으로 결론 내리기 어려운 쟁점을 PM이 자문 형태로 정리하여 회의에 상정 |

**프롬프트 (입력한 내용 요약)**

레포 전체 문서와 M3 양식(평가 루브릭 포함)을 첨부하고, 통합 과정에서 식별된 쟁점 3건의 처리 방안과 근거를 요청함: ① 설계 패턴 산출물이 별도 문서로 없는 상황에서 §8을 어떻게 작성할지 ② 인스펙션 결과가 기록으로 남지 않은 상황에서 §10을 어떤 형식으로 정리할지, 이때 미갱신 상태인 `requirements.md` 원본을 제출 전에 고칠지 ③ 일정 준수율 30%라는 낮은 수치를 어떻게 표기할지.

**AI 응답 요약**

① 패턴 문서를 소급 생성하지 말고 프로토타입에서 실제로 일어난 `MockAnalyzer`→`AIAnalyzer` 교체를 Strategy 패턴 적용 사례로 §8에 일원화하되, 적용 축은 `FactCheckAPI`가 아닌 실제 교체가 발생한 `Analyzer` 인터페이스로 잡을 것 (양식 §8의 "실제 적용 내용" 기준 근거). ② §10은 역할 교환 방향 4개별 결함·심각도·수정/미수정/수정 불필요 판정·사유 표로 정리하고 미수정 결함은 사유 병기할 것 (루브릭상 미수정 사유 기록이 우수 기준). ③ 준수율은 수치 조정 없이 의존성 연쇄 지연이라는 원인 분석과 §13 개선 방안으로 연결할 것 (루브릭이 '지연 원인 분석'을 '지연 없음'보다 높게 평가).

**비판적 검토**

- ②에서 AI의 초기 자문은 `requirements.md` 원본을 제출 전에 즉시 갱신하는 안이었으나, 그렇게 하면 §10에 기록될 "부분 수정(원본 갱신은 후속)" 판정과 원본 상태가 어긋난다는 점을 재질의함 → AI가 정합성 문제를 인정하고 "원본 미수정 유지 + 보고서 최종본 반영 + 후속 갱신 명시" 안으로 수정 제시함. 자문도 1차 답변을 그대로 쓰면 문서 간 모순이 생길 수 있음을 확인.
- 자문 3건 모두 팀이 그대로 수용하지 않도록 검증 절차를 분담함: ①은 설계자가 적용 전·후 다이어그램과 `main.py` 대조 + 개발자가 교체 이력 기준 시그니처 동일성·호출부 무수정 확인 — 이 검증 과정에서 최종 커밋본 코드에 `Analyzer` 인터페이스·`MockAnalyzer` 등이 유실되어 있음이 발견되어, §8 서술과 코드를 일치시키는 정합화 커밋(§10 미수정 결함 #4·#5 원본 상태는 유지)이 별도 결정됨. ②는 QA/보안이 6/4 검토 당시 결함 6건의 누락 여부 대조, ③은 전원이 양식 루브릭 원문 확인.

## 성찰 일지

**AI 활용 방식의 변화**

M1에서는 AI를 "양식 변환기"로 썼다면, M3에서는 "통합·검증 도구"로 쓰게 되었다. 특히 흩어진 산출물(문서 8종 + 회의록 3건 + 코드)을 한 번에 제공하고 교차 대조시키는 방식은 PM의 통합 작업 시간을 크게 줄였고, 사람이 놓치고 있던 문서 간 불일치(주차 라벨 2건, 경로 오기, 6/1 설계 검토 기록 부재, 보고서 서술-코드 불일치)를 드러내 주었다. 나아가 팀이 자체적으로 결론 내리기 어려운 SW공학 쟁점(패턴 문서 부재, 인스펙션 기록 형식, 지연 표기 수위)은 자문 형태로 AI 의견을 받아 회의에 상정하는 방식이 효과적이었다 — 단, 자문은 반드시 역할별 검증을 거쳐 채택했고, 채택 사실 자체를 회의록과 로그에 남겼다.

**확인한 한계**

12주차 API 오류 해결에서 AI는 실행 환경의 상태를 모른 채 해결책을 발산적으로 제시했고, 이를 그대로 따라가다 시간을 소모했다. PM으로서 배운 것은 AI 활용에도 일정 관리가 필요하다는 점이다 — 시도 횟수의 상한을 정하고, 일정 횟수 실패 시 사실 확인(모델 목록 조회 등)으로 전환하는 기준을 두어야 한다.

**개선 계획**

① 다음 프로젝트에서는 주차 라벨 같은 메타 정보를 작성 시점에 날짜 기준으로 통일하는 규칙을 초기에 정한다. ② AI에게 통합 검토를 시킬 때 "모순 발견 시 보고" 지시를 표준 프롬프트에 포함한다. ③ 환경 의존 오류는 시도 2회 실패 시 사실 확인 단계로 전환하는 팀 규칙을 유지한다.
